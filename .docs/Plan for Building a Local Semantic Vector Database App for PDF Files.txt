Plan for Building a Local Semantic Vector Database App for PDF Files

This plan outlines the steps to build an application that takes PDF files as input and outputs a semantic vector database stored locally on your PC.

Phase 1: Environment Setup and Dependency Installation
 1 Programming Language: Python
 2 Virtual Environment: conda
 3 Required Libraries: pandas numpy scikit-learn PyPDF2 nltk

Phase 2: PDF Processing and Text Extraction
 1 PDF File Input:
    • Implement GUI to allow users to select or specify the PDF files they want to process. Use either Tkinter or PyQt
    • Handle multiple file inputs (e.g., selecting a folder containing PDFs).
 2 PDF Parsing and Text Extraction:
    • Use the PyPDF2 library to open and read PDF files.
    • Extract text content from each page of the PDF.
    • Handle potential issues like password-protected PDFs or corrupted files.
 3 Unit Test for Phase 2:
    • Implement unit tests for PDF Processing and Text Extraction. Be sure to include tests for edge cases.

Phase 3: Text Preprocessing and Cleaning
 1 Tokenization:
    • Split the extracted text into individual words or tokens using nltk.word_tokenize.
 2 Stop Word Removal:
    • Remove common words (e.g., "the", "a", "is") that don't carry much semantic meaning using nltk.corpus.stopwords.
 3 Punctuation Removal:
    • Remove punctuation marks.
 4 Lowercasing:
    • Convert all text to lowercase for consistency.
 5 Lemmatization:
    • Reduce words to their base or root form using nltk.stem.WordNetLemmatizer. This helps group similar words together (e.g., "running", "runs" -> "run").
 6 Chunking:
    • Split PDF documents into smaller chunks preferably by delineation created in the table of contents (e.g. "1.0 [Section Title]", "1.1.1 [Sub-section Title]", etc.) before vectorization to improve performance and capture more granular semantic information. If table of contents is not present, then delineating by formatting of the PDF (i.e. using the PDF indent structure, use of number-text-new line-text, etc.) is an acceptable alternative. 
 7 Users Preprocessing Step Customization:
    • Stop word lists:
      • Allow users to add or remove stop words.
    • Stemming vs. lemmatization:
      • Offer a choice between stemming and lemmatization.
    • Vectorization method: 
      • Include options like word embeddings (e.g., Word2Vec, GloVe) in addition to TF-IDF.
 8 Unit Test for Phase 3:
    • Implement unit tests for Text Preprocessing and Cleaning. Be sure to include tests for edge cases.

Phase 4: Text Vectorization
 1 Vectorization Method:
    • TF-IDF (Term Frequency-Inverse Document Frequency)
 2 Implement Vectorization:
    • Use sklearn.feature_extraction.text.TfidfVectorizer to convert the preprocessed text into numerical vectors.
 3 Store the Vectorizer (note: decision on Phase 5-1 may impact this step):
    • If using FAISS, I am not sure the particulars of how FAISS is stored.
    • If using Pickle, save the trained TfidfVectorizer object (for reuse later without retraining).
 4 Dataset Resource Management for Large Datasets:
    • Utilize techniques to improve search performance for large datasets. For example (not exhaustive):
      • Dimensionality Reduction (e.g., PCA, SVD)
      • Approximate Nearest Neighbor Search (e.g., FAISS, Annoy) 
 4 Unit Test for Phase 4:
    • Implement unit tests for Text Vectorization. Be sure to include tests for edge cases.

Phase 5: Vector Database Creation and Storage
 1 Choose a Vector Database Format: 
    • FAISS (preferred; 1st choice) 
    • Pickle (also acceptable; 2nd choice)
 2 Store the Vectors:
    • If using FAISS, create an index and add the vectors to it.
    • If using Pickle, save the vectorized data (e.g., a NumPy array or SciPy sparse matrix) to a .pkl file.
 3 Metadata Storage:
    • Store metadata associated with each document (e.g., filename, title, author, etc.) alongside the vectors for easier retrieval and filtering.
 4 Unit Test for Phase 5:
    • Implement unit tests for Vector Database Creation and Storage. Be sure to include tests for edge cases.

Phase 6: Similarity Search
 1 Implement a Search Function (to search for similar documents based on a query):
    • Implement a search function that allows users to input either "keywords" or "full sentences" as queries. Preprocess queries using the same steps as documents, and vectorize them using the saved TfidfVectorizer.
    • Preprocesses the query in the same way as the documents.
    • Vectorizes the query using the saved TfidfVectorizer.
    • Performs a similarity search (e.g., using cosine similarity) against the vector database.
    • Returns the most similar documents.
 2 Unit Test for Phase 6:
    • Implement unit tests for Similarity Search. Be sure to include tests for edge cases.
